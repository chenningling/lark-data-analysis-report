## 推荐 Base 子表

每个分析任务创建一个新的 Base。表名应保留步骤顺序：

| 表名 | 用途 | 最小字段 |
| --- | --- | --- |
| `00_任务规划` | 原子化计划与过程追溯 | `步骤ID`, `关联问题ID`, `关联章节ID`, `关联图表ID`, `步骤名称`, `分析问题`, `输入来源`, `处理方法`, `输出表`, `状态`, `口径/假设` |
| `01_数据字典` | 数据源结构和质量画像 | `文件`, `Sheet`, `字段`, `推断类型`, `缺失数`, `缺失率`, `唯一值数`, `样例值`, `业务含义` |
| `02_清洗记录` | 清洗和转换日志 | `规则ID`, `字段`, `规则`, `影响行数`, `原因`, `产出表` |
| `03+_中间结果_*` | 中间计算结果 | 字段按每步计算结果设计 |
| `90+_图表数据_*` | 仪表盘就绪结果表 | 维度字段、指标字段、排名/占比/周期字段 |
| `97_图表注册表` | 图表与章节、仪表盘、图片的绑定关系 | `图表ID`, `章节ID`, `图表名称`, `图表类型`, `来源表`, `维度字段`, `指标字段`, `筛选条件`, `仪表盘组件`, `图片文件`, `报告块ID` |
| `98_分析章节规划` | 动态报告章节规划 | `章节ID`, `问题ID`, `步骤ID`, `图表ID`, `报告块ID`, `章节标题`, `决策问题`, `分析方法`, `证据表`, `短答案`, `置信度`, `限制`, `建议动作` |
| `99_结论索引` | 报告证据索引 | `结论ID`, `章节ID`, `图表ID`, `结论`, `证据表`, `证据字段`, `仪表盘组件`, `置信度`, `建议动作`, `优先级` |

## 分析原则

- 报告章节先由用户问题决定，再由字段可用性决定；不要先套固定模板。
- 每个正文分析章节绑定一个业务问题、一张证据表、一张图和一个行动建议。
- 每个正文分析章节绑定一个 `SECxx`，并通过 `97_图表注册表` 绑定 `CHxx`、仪表盘组件和图片文件。
- 每个核心指标首次出现时写清指标契约：粒度、分子/分母、过滤条件、时间范围。
- 每个转换步骤都应能从前置表或本地脚本复现。
- 仪表盘结果表优先使用长表结构：每行对应一个维度/时间桶/指标。
- 比率和占比必须保留分母。
- Top-N 图表应保留排序或排名字段。
- 在数据字典或数据源概览表中保留原始文件名和工作表名。
- 当缺失值、异常范围、重复键或时间粒度不一致影响解释时，必须写明限制。

## Base 写入模式

优先使用 `scripts/publish_to_lark.py` 发布，降低飞书接口细节导致的失败概率：

1. 在本地执行分析。
2. 为每张过程表、结果表和结论表生成 CSV。
3. 生成报告 Markdown 和图表图片。
4. 生成 `publish_manifest.json`。
5. 运行 `python3 scripts/publish_to_lark.py --manifest ./publish_manifest.json --cwd "$(pwd)"`。
6. 如失败，保留 state 文件重跑；具体处理见 `references/recovery.md`。

如果必须手工发布：

1. 创建 Base。
2. 先创建所有计划中的表和简单字段。
3. 写入 `00_任务规划` 和 `01_数据字典`。
4. 以每批不超过 200 行的方式写入中间记录和结果记录。
5. 创建 `97_图表注册表`、`98_分析章节规划` 和 `99_结论索引`。
6. 基于结果表创建仪表盘组件。
7. 创建飞书云文档报告并将图表插入对应章节。

## 完成清单

- 报告回答了用户的分析目标。
- 每个主要结论都引用至少一张 Base 表；相关场景下还应引用图表/仪表盘组件。
- 每个正文分析章节都有对应图表，或明确说明为什么该章节不适合图表。
- 每个 `SECxx` 都能回溯到 `Sxx`、证据表、`CHxx`、仪表盘组件、图片文件和 `RBxx`。
- 每个 `CHxx` 都出现在 `97_图表注册表`、`dashboard.blocks` 和 `doc.blocks`。
- Base 包含任务规划、数据源画像、计算过程、仪表盘就绪结果表、分析章节规划和结论索引。
- 最终回复同时包含飞书文档链接和 Base 链接。
